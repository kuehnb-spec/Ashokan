Ashokan Roadmap & Feature Design

Brainstormed July 7, 2026 · Brant Kuehn & Claude. A living document — edit it in Ashokan.

Guiding principle for every feature: the document must remain a plain, self-contained HTML file that renders correctly in any browser, with or without Ashokan. No sidecar files, no proprietary blobs, no lock-in — ever.

1. Review mode — tracked changes & comments ✅ shipped July 7 (sidebar & threads later)

The crown jewel, and the feature no other editor can offer in an open format. The trick: HTML already has semantic elements for tracked changes. <ins> and <del> are standard HTML — every browser renders insertions underlined and deletions struck through. So a document with pending edits is still just an HTML file, reviewable in Chrome by someone who has never heard of Ashokan.

  • Suggesting mode: a ProseMirror plugin intercepts every edit and rewrites it as <ins data-author data-ts> / <del> markup instead of applying it directly.
  • Comments: the commented range is wrapped in <mark data-comment-id> with the comment text mirrored into its title attribute — so even in a plain browser, hovering shows the comment. Threads/authors/timestamps live in a small JSON island at the end of the body.
  • Review toolbar: Editing / Suggesting / Viewing mode switch, previous/next change, Accept, Reject, comment sidebar on the right (Word-style).
  • The killer workflow: Claude Code or Codex proposes edits to a memo as ins/del markup; Brant opens it in Ashokan and accepts or rejects each change with a click. Human-in-the-loop document collaboration with AI agents, in an open format.

2. Images & tables — full word-processor competence ✅ shipped July 7

Images

  • Insert via toolbar/menu, drag-and-drop from Finder, or paste from clipboard.
  • Default: embed as data URI — keeps the document one self-contained file (the whole point of HTML docs). Option per-image to store alongside the file instead; warn on very large embeds.
  • Click to select: drag-handles for resize, alignment controls (inline / float left / float right / centered block), and a caption button that wraps in <figure>/<figcaption>.

Tables

  • Insert via a Word-style grid picker in the toolbar.
  • Row/column add & delete, merge/split cells, header-row toggle — prosemirror-tables supports all of it; we just expose commands.
  • Draggable column widths (the columnResizing plugin, CSS already stubbed in).
  • Fidelity: preserve <colgroup> and <caption> (currently dropped inside tables).

3. PDF export — print-quality pagination ✅ shipped July 7 (page numbers still to come)

WebKit already knows how to print HTML; the craft is in the pagination stylesheet Ashokan injects at export time:

  • break-inside: avoid on images, figures, tables, and code blocks — nothing splits across a page.
  • break-after: avoid on headings — no orphaned headings at page bottoms.
  • orphans/widows control for paragraphs; @page for paper size and margins.
  • Native print pipeline (NSPrintOperation) for headers, footers, and page numbers.
  • Export sheet: paper size, margins, page numbers on/off, then straight to a PDF that looks like it came from a word processor.

4. Recent files & launcher ✅ shipped July 7

Open Ashokan → a welcome window (when no document is open): a grid of recently edited files, each card showing a live-rendered thumbnail of the document, the filename, and beneath it the dimmed file path and last-opened date. Click to open. Thumbnails are WKWebView snapshots cached by path + modification time, so the grid is instant.

  • File > Open Recent already works (wired this session).
  • Later: a QuickLook thumbnail extension so HTML documents get real previews in Finder and Spotlight too.

5. iOS / iPadOS companion

The architecture bet already paid for this: the ProseMirror editing core doesn't know it's on a Mac — it runs in any WKWebView. The iOS app is a SwiftUI DocumentGroup shell around the same JS bundle.

  • Sync = files, not a service. Keep documents in iCloud Drive; the Files integration ("open in place") lets both apps edit the same file with no sync protocol, no server, no account.
  • Phone/iPad get view + edit + save, local or iCloud; the Mac stays the power tool.

Learnings from FrankenMarkdown (studied July 8 — Jeff Emanuel's Markdown→HTML/PDF engine)

Different species (headless compiler vs. our editor), but several concepts transfer. In priority order:

  • GFM task lists — agents write - [ ] checklists constantly; Ashokan should render them as real, clickable checkboxes (currently inert islands). Cheap, high value.
  • PDF export polish — set PDF title/author metadata; optional justified body with hyphenation (-webkit-hyphens in print CSS); document outline/bookmarks from headings (harder: WebKit print doesn't emit them; PDFKit post-process someday).
  • Dark-mode-aware default theme — our default/new-doc theme is light-only; add prefers-color-scheme support like FMD's output.
  • Mermaid rendering — FMD renders mermaid fences as SVG in HTML and PDF. Agent docs are full of mermaid; rendering (display-only, source preserved) would be a killer feature. Cost: ~1.5 MB bundled mermaid.js.
  • Display-only syntax highlighting for code blocks (~15 languages) — view-layer decoration, never written into the file.
  • Agent-first CLI — FMD's capabilities --json/robot-docs/stable-exit-codes contract is the right shape. Ashokan analog: a small headless CLI (ashokan export file.html --pdf) so agents can produce PDFs without the GUI, plus a machine-readable capabilities block in AGENTS.md.
  • Fragment-URL sharing — FMD encodes whole documents into URL fragments (deflate+base64url, nothing touches a server). When the Ashokan website exists, a static viewer page could accept #zdoc= links: instant private doc sharing. Website-phase idea.
  • Timestamp determinism — FMD guarantees identical output bytes; our only nondeterminism is data-ashokan-ts on suggestions. Consider an option to omit timestamps for git-diff-friendly documents.

Also on the someday list: find & replace across both panes, source-pane syntax highlighting, document outline sidebar, columns/layout helpers, and a Developer ID–signed release build.
