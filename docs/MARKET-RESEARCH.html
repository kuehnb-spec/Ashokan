Ashokan Market Research Omnibus — FINAL (Rounds 1+2 Merged)

A native macOS WYSIWYG editor for HTML/Markdown documents in the agent era

Synthesized July 2026 from: a 48-app landscape survey, 14 deep-dive competitive reports, three cross-community wishlist mining passes (round 1); plus an adversarial verification pass on the top 10 recommendations, five frontier scans (agent-doc point tools, canvas/artifacts complaints, docs-as-code, academic review, App Store review mining), four feature-spec studies (Word/Litera compare, Google Docs suggesting, VMark MCP, Typora/PowerPaste paste), and a completeness critique (round 2).

1. Executive Summary

Round 2 attacked the round-1 conclusions instead of extending them. The headline: the thesis held, but four of ten flagship recommendations were downgraded, one supporting citation was found to be effectively wrong, and the critic identified that the entire program rests on secondary sources with zero primary user contact. What changed and what didn't:

1. Lossless round-trip remains the moat — and round 2 found the competitor's own docs conceding the gap. VMark removed its tracked-changes MCP tools and now literally tells users "if you want to review every change, run an external git workflow." Nimbalyst shipped and had to fix a stale-buffer-clobbers-agent-edits bug. The canvas-complaints scan found ChatGPT Canvas's #1 documented pain is silent overwrite with no undo ("The content I was working on is gone. Completely unrecoverable." — https://community.openai.com/t/canvas-overwrites-work-without-warning-major-issue-for-creators/1231713), and its #2 is broken human↔AI sync (the model can't see the user's manual edits — https://community.openai.com/t/assistant-cannot-see-and-access-my-edits-in-canvas/1155732). Ashokan's file-on-disk, agent-reads-what-you-see architecture solves both by construction. App Store review mining independently confirmed data loss is the single most damaging 1-star pattern in the category (iA Writer, Bear), and that blaming the platform in responses makes it worse.

2. The category-rush finding is now quantified: at least a dozen point tools for reading/reviewing agent-produced Markdown shipped in ~18 months (md-redline, md-review, redline-marker, plannotator at 6.9k stars, revdiff, markdossier, Proof, MarkViewer, MacMD Viewer, OpenMark, OpenKnowledge). The category is validated and crowding fast. Crucially: (a) the traction leaders all close the loop back to the agent — annotation that becomes an agent prompt is the killer workflow; (b) distribution happens through the Claude Code plugin/hook ecosystem, not HN front pages (plannotator's repeated Show HN posts got 2–5 points while the repo accumulated 6.9k stars); (c) annotation storage is fragmenting into four incompatible ad-hoc formats with admitted anchoring fragility (~85% accuracy after revisions) — Ashokan's in-document standard ins/del/mark markup is the strongest differentiator in the field; (d) $10–20 one-time pricing is proven for viewers alone (MacMD Viewer $19.99, OpenMark $9.99), so Ashokan can price above it.

3. OpenKnowledge's Show HN (381 points, June 2026) is the positioning script. The largest relevant HN audience asked for exactly: local Ollama integration, a comments/suggestions/red-lining UI, lossless preservation during agent edits, and native (non-Electron) speed — the four things Ashokan already ships. Top complaints: marketing showing features the app lacked, and Electron sluggishness. (https://news.ycombinator.com/item?id=48675435)

4. Verification changed conclusions — honestly reported. CONFIRMED at high confidence: bulk accept/reject + clean-preview toggle; external-change detection; Mermaid/KaTeX rendering. CONFIRMED at medium confidence: compare/diff (but reframed as a safety-net, not hero feature), MCP agent-edit intake (but with a mandatory file-watch fallback, since agents default to plain file writes), smart paste (scoped to agent-relevant sources, not legacy-Word fidelity). WEAKENED: dual-pane click-to-source sync (the Whisk "most-requested" claim traced to a two-person forum thread; the Nimbalyst cursor-loss claim appears wrong — its docs say cursor position is preserved), comment threading (no in-file HTML standard for threads exists; the "standards-readable threading" framing overpromised), pre-send scrub (the marquee metadata-leak incidents are all 20+ years old and concern hidden markup, which Ashokan by design doesn't have), and paste-attribution (iA Authorship has no adoption evidence and Ashokan's core audience has no paste step). Nothing was REFUTED — which the completeness critic correctly flags as itself a confirmation-bias warning (see §8).

5. The unexamined risks are now named. No primary user research exists; no willingness-to-pay data for Ashokan's own segment; no demand sizing; the load-bearing "HTML shift" macro-thesis rests on one viral essay; agent compliance with Ashokan's markup conventions has never been empirically benchmarked; and the existential scenario — Anthropic/Cursor shipping rendered diff-review natively — has never been analyzed. All are cheap to close (§8) and should gate the bigger builds.

2. The Landscape (condensed from round 1)

2.1 Classic WYSIWYG HTML editors — dead, and why

Dreamweaver (abandoned 2021), BlueGriffon (dead 2024), KompoZer/Nvu, Expression Web, SeaMonkey Composer. Four causes, none applying to Ashokan: engine maintenance burden (BlueGriffon's author couldn't maintain a Gecko fork alone — https://en.wikipedia.org/wiki/BlueGriffon; Ashokan rides Apple-maintained WebKit); rendering drift (Dreamweaver's fake renderer torched trust; Ashokan renders with the real engine); the responsive web broke the single-canvas illusion for web pages — documents have a substantially single true rendering; vendor exit. Demand never died — "KompoZer alternative" searches and HN nostalgia threads persist. The market collapsed on the supply side.

2.2 Mac Markdown editors — crowded, but the WYSIWYG+review corner is empty

Typora ($15, the WYSIWYM bar, no review workflow), iA Writer (native polish + Authorship provenance prior art), Obsidian (vault gravity), Bear/Ulysses (proprietary-storage anti-pattern), MarkEdit (4MB native), MacDown/MarkText (abandoned; MarkText's 50k stars prove free-WYSIWYG demand — though the critic notes stars ≠ retained users), OpenMark ($9.99, racing for the SEO ground). No player combines WYSIWYG + HTML-native + tracked changes + agent workflow. Typora's 2026 "Open in Typora" VS Code extension and Obsidian's CLI show the segment drifting toward Ashokan's audience without owning the review job.

2.3 Collaborative review / word processors

Word defines the vocabulary (Track Changes, markup views, Compare — 72% of legal pays Litera for a better Compare); Google Docs defines the modern interaction (Suggesting mode margin cards). Fresh graveyard: Quip, Dropbox Paper, Almanac ("GitHub for docs," 4.7/5 stars, dead Jan 2025 because it demanded whole-team migration). Every cloud doc's review markup dies with the vendor; plain HTML ins/del on disk cannot be end-of-lifed.

2.4 Web editor frameworks

CKEditor 5 (reference track-changes, extortionate sales-led pricing), TinyMCE (MIT→GPL backlash), Tiptap (agent-edit review as $49–999/mo premium; free-tier removal burned goodwill), ProseMirror/Lexical (structurally lossy — Tiptap's own FAQ concedes non-conforming content "will be lost," https://tiptap.dev/docs/guides/faq). The framework world cannot do lossless arbitrary-HTML round-trip; enterprises pay hundreds monthly for exactly the review feature set Ashokan ships natively.

2.5 AI-document tools — the gold rush

ChatGPT Canvas / Claude Artifacts generate the document supply and define mainstream expectations — and their documented failures (silent overwrite, broken sync, full-regeneration instead of targeted edits, export friction, length ceilings, outages) are Ashokan's positioning script (§4.K). Direct neighbors: Nimbalyst (beloved rendered diffs, lossy round-trip, 322 open issues), VMark (clean MCP, no review step — conceded in its own docs), plus the round-2 point-tool catalog (§3.14). Nobody combines: native app + HTML-native WYSIWYG + lossless round-trip + standards-based review + local AI. That intersection is Ashokan.

3. App-by-App Capsules (with round-2 corrections)

3.1 Adobe Dreamweaver (abandoned)

Roundtrip HTML won ~66% of pros by 1999 (https://thehistoryoftheweb.com/dreamweaver-got-name/); split code/design view with click-to-source was the most durably loved UI pattern; died of the responsive web + Adobe neglect. Round-2 correction: the "loved split-view" evidence is 15+ years old, from a web-developer audience — still valid as pattern precedent, no longer valid as demand proof for Ashokan's segment. Lesson: lead with the lossless wedge, render with the real engine, stay a document editor.

3.2 BlueGriffon (dead 2024)

Clean output, true Gecko fidelity; died of solo browser-engine maintenance; paywalled manual bred resentment. Round-2 correction: its "top missing feature = split view" evidence comes from old mailing-list chatter about a crippled plugin in a defunct product — stale, tiny-niche signal, downgraded. Lesson: platform engine, don't paywall learning, communicate cadence.

3.3 Tumult Whisk (alive, barely evolving)

Instant per-keystroke WebKit preview; 20+ years of paying customers at $29.99 validates the native-utility model; percentage-based scroll sync broken for its entire life. Round-2 correction: the claim that element-level click-to-source sync is Whisk's "most-requested unsolved feature" traces to ONE forum thread with two participants (https://forums.tumult.com/t/use-of-html-wysiwyg/23560), and Whisk already ships proportional Scroll Sync. Real, but one user's say-so plus a founder's "good feature request." Lesson stands, evidence downgraded.

3.4 Pinegrow (alive, plateaued)

Real files on disk, no lock-in, profitable since 2014; WYSIWYG text editing is its weakest surface (years-long unanswered forum megathread). Lesson: native-quality selection/paste/undo is make-or-break; answer your forums.

3.5 Typora (alive, the WYSIWYM bar)

Category-defining inline WYSIWYG; $15 one-time sustainable; history of data-loss bugs (0-byte saves) and silent Markdown rewriting it had to revert under user anger. Round-2 paste-spec study (§5, rec 8) extracted its full paste architecture — HTML-flavor-first with plain-text fallback, mode-dependent behavior, Cmd+Shift+V escape hatch, zero prompts — plus its documented failure catalog (line-break normalization, code-editor paste mangling: typora-issues #5467, #6305, #5664). Lesson: never rewrite untouched markup; paste predictability without dialogs is a praised design choice.

3.6 iA Writer (alive, the provenance pioneer)

Best-in-class native polish; Authorship (auto-detect pasted AI text, "Paste Edits From" delta attribution) praised in concept, jeered in format (proprietary hidden metadata block leaking through Hugo/Jekyll — https://www.andreagrandi.it/posts/ia-writer-authorship-brilliant-feature-problematic-implementation/). Round-2 correction: no adoption data for Authorship exists; HN sentiment on the feature is split; the most detailed independent reviewer admits he barely uses it. The format lesson (standard markup wins) survives fully; the demand claim for paste-attribution is WEAKENED. App-store mining adds: iA's sync data-loss 1-stars are savage, and developer responses blaming Apple/iCloud made reviews worse.

3.7 OpenMark (new, the SEO race)

$9.99, native, zero-config Mermaid/KaTeX, growing quietly via keyword-targeted /compare/ pages. Round-2 correction: all discoverable "praise" for OpenMark is its own SEO blog content; no independent Reddit/HN reviews found. The distribution playbook remains worth copying; the "praised for it" citations are downgraded to vendor marketing.

3.8 Microsoft Word (dominant incumbent)

Track Changes is the professional lingua franca; performance collapses on long documents (Microsoft's own KB); Copilot edits didn't arrive tracked until April 2026 (still beta, Windows-only, can't accept/reject — https://windowsforum.com/threads/copilot-in-word-adds-track-changes-comments-for-auditable-enterprise-reviews.413287/). Round-2 spec study extracted the complete Compare/Combine feature model (§5, rec 7) and confirmed the by-reviewer bulk-accept workflow is actively documented and demanded (a 2026 Microsoft Q&A thread asks for finer bulk accept — https://learn.microsoft.com/en-au/answers/questions/5518510/). Notable: Word cannot attach a comment thread to a tracked change — a real gap.

3.9 Google Docs (dominant)

Suggesting mode's margin-card loop moved editorial workflows off Word. Round-2 spec study (§5, rec 2) extracted the full interaction model: mode as sticky document state, per-author colors, formatting-only suggestions as first-class, card anatomy, threads-on-suggestions, three action granularities, and the three preview modes (SUGGESTIONS_INLINE / PREVIEW_SUGGESTIONS_ACCEPTED / PREVIEW_WITHOUT_SUGGESTIONS — https://developers.google.com/workspace/docs/api/how-tos/suggestions). Documented complaints Ashokan can exploit: 3–5s/keystroke lag in suggesting mode (https://discussions.apple.com/thread/255075128), the granularity gap (no multi-select, no filter-by-author, no accept-within-selection), suggestions silently omitted from downloads, and unremovable timestamps on shared suggestions. Google eventually shipped native Accept all / Reject all + previews after years of add-on workarounds — incumbent capitulation validating rec 1.

3.10 Almanac (dead Jan 2025 — the validated-demand corpse)

Branch/compare/merge and per-edit accept/reject called "killer features" at 4.7/5; died of the all-or-nothing migration tax despite ~$50M raised. Round-2 addition: Almanac's death is also counter-evidence — a loved compare feature did not sustain a business, so "beloved" ≠ standalone commercial value. Lesson: never demand migration; review artifacts must be readable without the app.

3.11 CKEditor 5 / TinyMCE / Tiptap (frameworks)

CKEditor: reference track-changes implementation (comments-on-suggestions, bulk accept/decline) at €15k/year quotes. Tiptap: sells agent-edits-as-reviewable-suggestions as its premium headline; its pricing/lock-in complaints are an explicit opening for a local open-format alternative (https://eddyter.com/blogs/tiptap-pricing-explained-2026). TinyMCE PowerPaste: the reference for paste-cleanup depth — and the reminder that full legacy-Word fidelity is a multi-year swamp Ashokan should explicitly not enter.

3.12 Nimbalyst (closest thesis-neighbor)

Rendered red/green diffs are the most-loved feature in the niche; lossy round-trip is the most damning bug class; 322 open issues at ~1,100 stars = scope creep cracking quality. Round-2 corrections: (a) its release notes confirm it shipped and fixed the exact stale-editor-clobbers-agent-edits bug (https://github.com/Nimbalyst/nimbalyst/releases) — validating rec 4; (b) the round-1 claim that its source/WYSIWYG toggle loses cursor position appears wrong — its docs describe a source-only Monaco editor with no visual toggle and state cursor position is preserved (https://docs.nimbalyst.com/visual-editors-powered-by-ai/code-editor). Citation retracted.

3.13 VMark (the MCP ergonomics reference — now fully spec-mined)

Round 2 read VMark's source, ADRs, and issue tracker. Key extractions (full requirements in §5, rec 5): the 5-tool composite surface (cut from 12 tools/76 actions, citing tool-selection accuracy dropping 95%→71% on crowded surfaces); revision-token optimistic concurrency rated "skipping this ships a data-loss bug"; automatic per-mutation checkpoints with restore UI; structured save-outcome envelopes; the path guard it had to retrofit (PR #1080); one-click per-assistant installers with path-mismatch repair — its most polished UX. The headline finding: VMark deleted its suggestions feature and its docs now say "use git" for change review — the exact gap Ashokan fills, conceded in the competitor's own documentation. Its issue tracker shows users judge MCP almost entirely on connection reliability and setup ergonomics, not tool semantics.

4. Cross-Cutting Themes

Carried from round 1 (still standing): A. true inline WYSIWYG with source escape hatch; B. just-open-a-file, no vaults; C. tables are the biggest Markdown pain; D. effortless images in one portable file; E. track changes is the Word feature MD/HTML tooling never got (git rejected as line-not-word); F. fast and native, not Electron; G. trustworthy PDF export with page control; H. don't touch what I didn't edit; I. legal-review pains as feature specs; J. the agent-document moment (HTML shift per HN 48071940 — now flagged as resting on thin evidence; see §8).

New from round 2:

K. The canvas paradigm's betrayals are the pull-off-Canvas emotional wounds. Users love conversational co-editing on a shared visible surface ("Google Docs met an AI") and judge tools on targeted-edits-vs-full-regeneration; they are betrayed by silent overwrite, invisible manual edits, burned token limits from whole-document re-emission, formatting-destroying copy-paste out, truncation on long documents, and outages. Every requested remedy (version history, diffs, sync, confirmation) is a request to make co-editing safer, not to abandon it. A whole extension ecosystem exists solely to liberate Canvas/Artifact content into local Markdown — users' desired end state is a local file they own. Positioning: "the canvas experience, without the betrayals," and "there is no export step; it's your file." (Sources: OpenAI community threads 1231713, 1155732; https://venturebeat.com/ai/chatgpts-canvas-now-shows-tracked-changes; https://www.aifire.co/p/detailed-comparison-for-interactive-tools-canvas-or-artifacts)

L. Docs-as-code's acknowledged bottleneck is SME/stakeholder review — and every bridge is cloud. Non-technical reviewers can't/won't use raw Markdown diffs; the dominant workaround is the lossy Google-Docs shuttle; preview deploys solve viewing but not commenting; GitBook/Mintlify/Fern/DraftView all bridge via hosted SaaS requiring accounts. The community even bends the source format around line diffs (AsciiDoc's sentence-per-line convention exists for diff hygiene). Ashokan's gap: a local, native reviewer where the rendered document is the review surface and the result is still the file in the repo. Vale's CI adoption (Datadog: 20k docs PRs/year) shows machine-assisted first-pass prose review is already accepted — Ashokan's local Ollama review is the same philosophy without CI. Review data in-file + git = an audit trail compliance buyers can read forever. (https://www.draftview.app/blog/docs-as-code-review-workflow-complete-guide, https://dacharycarey.com/2020/05/23/docs-as-code-workflow-the-missing-link-a-collaboration-tool/, https://www.datadoghq.com/blog/engineering/how-we-use-vale-to-improve-our-documentation-editing-process/)

M. Academic revision workflows are a ready-made feature map. Journals universally demand a marked copy + clean copy pair (https://www.elsevier.support/publishing/answer/how-do-i-submit-my-revision-with-tracked-changes); the dominant retroactive workflow is Word Compare on before/after versions; latexdiff — the de facto LaTeX diff — is notoriously fragile on exactly the structures Ashokan handles natively (tables, figures, math); Overleaf paywalls track changes and strands review data on downgrade; Word's Combine merges only two documents at a time and loses author attribution; the point-by-point response-to-reviewers letter is heavily manual. Concrete Ashokan hooks: one-click marked+clean export pair; a "latexdiff that just works" for HTML/MD; N-way feedback merge with preserved attribution; a revision-letter generator from comment cards. (https://dynamicecology.wordpress.com/2025/09/04/i-wish-journals-didnt-ask-for-ms-revisions-with-tracked-changes/, https://docs.overleaf.com/collaborating/track-changes)

N. App Store patterns: reviews are won on feel and lost on data loss. 5-star delight clusters on getting-out-of-the-way, raw speed, and responsive developers; 1-star death clusters on sync/data loss (and deflecting blame), silent rewriting of user files (Ulysses), pricing betrayal (the 2017 Ulysses subscription switch still poisons reviews a decade later), per-platform repurchase anger, and removing features in paid upgrades. Minimalism must not exclude table-stakes basics (in-document find, system accessibility settings, window-state restoration). Niche Mac utilities (Whisk) suffer thin review volume and "abandoned?" suspicion — actively prompt happy users for ratings and ship visibly regular updates. (https://apps.apple.com/us/app/ia-writer/id775737172?see-all=reviews, https://gizmodo.com/another-great-app-went-subscription-only-and-everything-1797802223)

5. FINAL RECOMMENDATIONS — Ranked Feature List

Evidence grades: [VERIFIED-STRONG] = survived adversarial verification at high confidence; [PLAUSIBLE] = survived at medium confidence, or well-corroborated but not adversarially tested; [WEAKENED] = survived but materially downgraded; [REFUTED] = none (itself a caution — see §8). [HAS] = already shipped. Effort: S/M/L. Ranked by evidence × impact ÷ effort.

1. [HAS — harden] Crash-safe autosave, atomic writes, local version snapshots — [VERIFIED-STRONG] (M)

Promoted to #1 by round 2: silent data loss is the single most severe documented pain in both the canvas world (Canvas overwrites, no undo, no history) and the App Store (iA Writer/Bear 1-star patterns), and it is the precondition for everything else. Requirements: autosave with local snapshots; atomic temp-file+rename writes; undo that survives AI operations; never respond to a data-loss report by blaming the OS or the agent. External-change handling (rec 4) is Ashokan's equivalent of the sync-loss trap — treat it as the top reliability scenario. Marketing: "your document is a real file that can never be silently destroyed." (https://community.openai.com/t/canvas-overwrites-work-without-warning-major-issue-for-creators/1231713)

2. [NEW] Bulk review operations + non-destructive preview modes — [VERIFIED-STRONG] (M)

Accept/reject: all / within selection / by author (human vs AI vs named agent) / by change type; plus three render modes over the existing markup: markup view, preview-as-if-all-accepted, preview-as-if-all-rejected (Docs' SUGGESTIONS_INLINE/PREVIEW_SUGGESTIONS_ACCEPTED/PREVIEW_WITHOUT_SUGGESTIONS; Word's No Markup/Original). Verification held everywhere it was attacked: Google capitulated and shipped native Accept all + previews after an add-on cottage industry; professional editors' workflow ("reject the few, Accept All the rest") depends on bulk ops across 10–30k revisions per manuscript; Docs users still beg for the granularity Google lacks (accept-within-selection, filter-by-author — https://support.google.com/docs/thread/95333134). This is table-stakes completion of shipped review mode, not a differentiator. Design requirements/cautions (from verification + Docs spec): the No-Markup trap is failure mode #1 — clean preview must be visually unmistakable (persistent banner), and save/export/copy while previewing must warn or explicitly resolve (a user handing "clean-looking" HTML with live ins/del to an agent corrupts the next pass); make bulk scope explicit (all vs shown vs selection vs author — Word's silent distinction burns users); every bulk op = one undo unit; by-author filtering requires consistent author attribution stamped on every ins/del including agent-emitted markup — document the attribute convention in the embedded agent instructions; decide and communicate whether bulk resolve touches comments (chronic Word confusion); define resolution order for nested ins/del; compute word count against the accepted-preview; UI should favor "accept all remaining" over blanket Accept All.

3. [HAS — harden and prove] Lossless round-trip as a tested, demonstrable guarantee — [PLAUSIBLE — load-bearing, unverified as macro-thesis] (M)

Unchanged from round 1: adversarial regression corpus (agent-generated HTML, giant tables, unknown markup, 5MB files) on every release; user-visible "what actually changed" diff after save. Round 2 added the sharpest marketing evidence yet (VMark's "use git" concession; Ulysses' silent-rewriting 1-stars; OpenKnowledge commenters demanding frontmatter/code-fence preservation). Caveat: the critic notes the HTML-shift demand thesis behind this rests on one essay + anecdote — the guarantee is still correct engineering, but measure the format mix (§8.7) before betting marketing entirely on HTML.

4. [NEW] External-change detection + conflict-safe saves (the two-writers feature) — [VERIFIED-STRONG] (M)

Verification failed to break it: Claude Code's own tracker is full of the human-editor-vs-agent write race seen from the agent side (#10437, #7883 "File has been unexpectedly modified"); Nimbalyst shipped and fixed the exact clobbering bug; Obsidian agent-plugins independently converged on multi-writer safety; Typora's no-seamless-reload complaint has been open since 2016. A file-based agent-collaboration editor without this fails its core use case on day one. Design requirements/cautions: auto-reload only on a clean buffer; dirty buffer → three-way merge or non-modal conflict UI (never a modal prompt, never silent reload); pre-write check mtime+content hash, write atomically; watch the parent directory (atomic saves change the inode and kill file-node watchers); suppress self-echo (ignore writes you originated); debounce/coalesce (agents write the same file many times in seconds); diff against normalized serialization, not raw bytes; keep external-change highlighting semantically distinct from review-mode ins/del — and decide explicitly whether an agent's external edit becomes an acceptable tracked change or an ephemeral highlight; preserve cursor/scroll/margin-card state across reload; handle delete+recreate as modification, not a scary "file deleted" flash. The change-highlighting layer is the differentiating second tier built on existing review markup; the detection/safety layer is table stakes.

5. [HAS — extend] MCP server: agent writes land as tracked changes by default — [PLAUSIBLE] (L)

CONFIRMED at medium confidence with a critical reframe: the durable core is "any agent edit becomes reviewable tracked changes in the document"; MCP is only one intake path. Agents default to plain Write/Edit on files and will bypass the MCP unless instructed — so the file-watch/diff-import path (rec 4 + rec 7's diff engine) is what makes the promise true for any agent. Demand is real and convergent (six independent builders shipped agent-edits-as-suggestions in ~12 months; Claude Code issues #31888/#40409 are Ashokan's exact audience demanding batch review of agent edits), but the niche is crowding (Simple Markdown Editor ships the identical pitch) and the free git-diff substitute caps willingness-to-pay for the code-adjacent segment. Requirements skeleton (from the VMark spec study — every element has a VMark precedent or fills a gap VMark documented):

  • ~5–6 composite tools, ~18 actions max: session (get_state incl. per-tab review state: kind, reviewMode, pending counts, author identity), workspace (with path guard + APPROVAL_REQUIRED), document (read/write/transform), selection (get/set), and ONE new review tool (list pending changes with stable change-ids, comment, optionally accept/reject behind an auto-approve setting). Don't build a parallel suggestions.* tool family — that's the shape that died at VMark; keep the agent-facing surface identical to normal writes and change only where the write lands.
  • Revision-token optimistic concurrency on every mutation; STALE carries current_revision; every user action including accepting a change bumps the revision; integration-test that stale writes are no-ops and that tracking is actually wired (VMark shipped it silently dead, #937).
  • document.read must offer view parameter: markup / as-if-accepted / as-if-rejected, so agents can reason without choking on review markup.
  • Server-side targeted-edit path (anchored patch) guaranteeing zero collateral edits — full-content agent rewrites are inherently lossy; version the patch schema from day one.
  • Pre-image checkpoints per mutation with a visible restore UI; structured save-outcome envelopes; exempt your own MCP writes from external-change detection (VMark shipped that bug twice, #975/#982).
  • Path guard scoped to workspace + open documents on day one (VMark retrofitted it, PR #1080); localhost-only with auth token; uniform error envelope with enumerated codes extended for review semantics (NOT_IN_REVIEW_MODE, CHANGE_NOT_FOUND, ALREADY_RESOLVED).
  • One-click per-assistant installers (Claude Desktop/Code incl. project .mcp.json, Codex, Gemini) with path-mismatch Repair, connection indicator, health dialog, and a --health-check flag — VMark's issue tracker proves users judge MCP on connection reliability and setup, not semantics.
  • Tool descriptions as mini-prompts teaching the review contract ("your writes appear as tracked changes; the user accepts or rejects; check pending state before assuming your edit landed").
  • Autonomy dial: two honest positions (always-suggest / auto-apply-with-undo-log), not a "safe/destructive" classifier that will misfire; rationale attaches at hunk/card level, not per-word (comment spam on a 200-edit pass is the documented failure).
  • Gate the build on the agent-compliance benchmark (§8.5).

6. [NEW] Zero-config Mermaid + KaTeX rendering — [VERIFIED-STRONG] (M)

Confirmed at high confidence, reframed as parity/table-stakes, Mermaid first: VS Code shipped built-in Mermaid preview in 1.121 (Feb 2026) after years of duplicate requests (https://github.com/microsoft/vscode/issues/293028); GitHub called it a top request since 2015; an entire validator/fixer ecosystem exists because agent output is dense with (frequently invalid) Mermaid. A mermaid fence rendering as plain text will read as a defect. Weak plank: the "OpenMark praised for it" citation is vendor SEO; math demand skews academic — KaTeX rides along, doesn't lead. Cautions: never render a blank box on parse error — graceful inline error with raw fence visible, plus a one-click "fix with local AI" hook into the Ollama pipeline; bundle a recent Mermaid, surface the version, plan update cadence (version drift = "renders on GitHub, breaks in your app"); the fence stays plain source in the file — render read-only, click-to-edit source, never serialize SVG back; support $...$, $$...$$, and ```math; securityLevel 'strict' (agent documents are untrusted input); lazy-load the ~2–3MB bundles only when needed; define behavior in HTML mode and PDF export (users expect the rendered diagram in the PDF).

7. [NEW] Compare/diff two documents into tracked changes — [PLAUSIBLE] (L)

Confirmed at medium confidence, reframed: a safety-net/recovery feature (what changed when redlines weren't produced), not the hero — the ecosystem is moving toward agents emitting tracked changes natively, making Compare the fallback path. Demand corroborated (evergreen "forgot to track changes → Compare" advice; new entrants launching in the legal-compare niche; 8-year-open GitLab word-diff-for-prose issues; a workaround cottage ecosystem). Scope: two files or file-vs-git-HEAD, word-level ins/del out, nothing more. It also unlocks recs 5 (diff-import intake), 15 (paste attribution), and the academic export pair. Key design requirements (condensed from the 22-requirement Word/Litera spec study): word-level default with character-level option; move detection as paired linked del.move-from/ins.move-to (accepting one side accepts both; minimum move-unit threshold); formatting-only changes as a third change type, toggleable — formatting-noise suppression is the #1 incumbent complaint; the standard toggle set mapped to HTML/MD (case, whitespace, tables, comments, link-href-only changes, generated content like TOCs → one collapsed "TOC updated" change); output IS a live reviewable document in Ashokan's existing markup, attributed to a configurable label, new-document by default (never mutate sources); pre-existing review markup in inputs handled deterministically (compare as-if-accepted by default, with prompt — Word's warning-dialog behavior); Combine (N-way merge preserving per-source attribution, surfacing conflicts instead of Word's silent pick-one) as the agent-era sequel; tables diffed structurally (align rows by similarity, never whole-table delete+insert — Word's most-cursed weakness and Litera's headline differentiator); list renumbering must not cascade-noise; hierarchical block-first alignment so one mismatched block never poisons the rest (Word's "runaway diff" failure); invisible changes described in words in the margin card; images compared by src/hash as one "image changed" card; change-list navigation with counts/filters/jump; rendering sets as CSS themes; and the self-check invariant neither incumbent exposes: accept-all reproduces exactly document B, reject-all exactly document A — verified automatically after every comparison. Entry points: File menu, workspace right-click two files, drag-onto-open-doc, and agent-invocable CLI/MCP. Cross-format (HTML vs MD of "the same" doc) via normalization to a common AST; PDF input explicitly out of scope for v1.

8. [NEW] Smart paste — [PLAUSIBLE] (M)

Confirmed at medium confidence, rescoped for Ashokan's audience: (1) rich HTML/web paste sanitized to clean document HTML; (2) spreadsheet-range paste into tables; (3) markdown-aware paste (detect raw MD / AI-chat clipboard and convert) — the highest-leverage item, under-weighted in round 1 (a 2026 cottage industry of paste-fixer tools monetizes exactly the ChatGPT/Claude-to-document friction). Deprioritize deep legacy-Word fidelity — that's TinyMCE's enterprise battlefield. Requirements (from the Typora/PowerPaste spec study): one deterministic rule — HTML flavor wins, plain-text fallback (REQ-1); source pane always pastes verbatim plain text (REQ-2); Cmd+Opt+Shift+V (AppKit standard) + Cmd+Shift+V alias in the Edit menu, plus a polarity-inversion preference (REQ-3); paste-as-plain-text means parse-as-Markdown-source, with an explicit literal option (REQ-4); whitelist sanitize pipeline for Word/web (strip mso-*, classes, styles; keep semantic structure) with mso-list nesting reconstruction (REQ-5/6); spreadsheet <table> → native table, TSV detection offer (REQ-8); special-case code-editor HTML — detect VS Code's syntax-colored flavor and prefer plain text into a code block; inside a code block, always plain verbatim (REQ-9 — the classic curse, and a beat-Typora opportunity for this audience); image data → assets folder beside the document with relative src, never base64 by default (REQ-10); recover Word-embedded images from native pasteboard flavors or insert visible placeholders (REQ-11); line-break normalization with a fixture corpus from Word/Docs/Safari/VS Code/Excel (REQ-12 — the most-cursed edge case); on copy, write HTML + RTF + plain-text-as-exact-source (lossless; feeds agents the real markup) (REQ-13); no prompts ever — one predictable default + shortcut + transient undo-able toast at most (REQ-14); staged pipeline architecture, each stage fixture-tested (REQ-15). Cleaning applies only to the pasted fragment — never triggers document rewrite.

11. [NEW — scoped down] Export-time review check — [WEAKENED] (S)

12. [NEW — scoped down] Pane sync: scroll sync + cursor preservation now; click-to-source later — [WEAKENED] (S–M)

The round-1 evidence largely collapsed (Whisk two-person thread; Nimbalyst claim contradicted; BlueGriffon stale; Dreamweaver 15 years old; VS Code gives this audience free scroll-synced preview already). What stops refutation: the pattern is ubiquitous — once you show two panes, unsynced panes read as a defect. Ship the cheap 80%: bidirectional scroll sync mapped by element/sourcepos anchors (not scroll percentage — the documented failure mode), cursor/selection preservation across pane toggles (mapped through edit history, not cached offsets), debounced with the initiating pane marked to avoid feedback loops. Defer full DOM-mapped click-to-highlight until real users ask. Source-mapping must be a read-only position index over original bytes — never a rewrite. Don't grow a DevTools inspector.

13. [NEW — scoped down] Comment replies + resolve-with-history, anchored to changes — [WEAKENED] (M)

Build scoped: replies + resolve/reopen on existing comments; let a thread anchor to a tracked-change element by stable id (never character offsets); minimal data-* convention (data-parent-id, data-resolved-by/at) documented in the embedded agent instructions — the agent-readability is the differentiator; resolve never deletes (hidden, toggleable, strip-on-export option); market it as "plain HTML your agent and browser can read, with a documented convention," NOT "standards-based threading." 6. Anti-Recommendations

Carried from round 1, all still standing: 1. don't become a site builder; 2. don't drift into knowledge management (no vaults/backlinks/graphs); 3. don't build agent orchestration (Nimbalyst's 322-issue cautionary tale; orchestration is Claude Code's job); 4. don't require projects/accounts/cloud/sidecar state (Almanac's death; every review artifact lives in the document); 5. don't build Lock Tracking theater — build Compare; 6. never auto-normalize/prettify/inject; 7. no proprietary provenance formats (iA Authorship's documented failure); 8. no metered/per-seat/bait-and-switch pricing, no paywalled docs (round 2 adds: the Ulysses subscription backlash persists in reviews a decade later; never ship a paid tier that drops something users had); 9. no real-time cloud co-editing; 10. don't go cross-platform early (but state the Mac-only stance up front — OpenKnowledge's thread shows loud Windows/Linux demand arrives immediately); 11. don't gate learning or go silent (release cadence is a retention feature; "abandoned?" is the niche's default suspicion).

New from round 2:

12. Don't store annotations in sidecar files or invisible markers. The point-tool field fragmented into four incompatible sidecar/marker conventions with admitted anchor drift (~85% accuracy). In-document standard markup is the differentiator — "your review can't drift from your document."

13. Don't build a parallel suggestions. MCP tool family.* That's the exact shape VMark shipped and deleted. Keep the agent-facing surface identical to normal writes; change only where the write lands.

14. Don't chase legacy-Word paste fidelity (Word Art, charts, field codes, revision marks). Define a supported paste matrix and punt the rest to plain text — even TinyMCE doesn't fully solve it after 20 years.

15. Don't ship an autonomy dial with a "safe vs destructive" classifier. One misclassified meaning-changing edit destroys trust. Two honest positions; heuristics later, conservative by default.

16. Don't let marketing outrun the app. The single harshest criticism in the OpenKnowledge thread was screenshots showing features the app lacked; and don't touch shell config or anything else without consent during install.

8. Research Gaps That Remain (from the completeness critic)

The verification round tested the feature recommendations; these program-level gaps remain open, each with a cheap closing action:

  1. Zero primary research. No target user has ever been interviewed or observed; every conclusion is secondhand about other products' users. → 5–8 recorded 30-minute sessions (recruit from r/ClaudeAI, Cursor/Anthropic Discords): hand them an agent-generated report and Ashokan, watch the first 10 minutes. One weekend; could reorder the entire top-5.
  2. No willingness-to-pay data for this segment. Competitor prices are known; what this buyer pays is not — and it determines scope ($9.99 can't fund the MCP + compare roadmap). → Fake-door landing page with two price anchors + 3 pricing questions in the interviews.
  3. No demand sizing. The niche is proven empty and the pains real, but nobody counted the people in it. → One day: Ahrefs volumes for the ~15 head terms in rec 16; macOS share of agent-coding users from public surveys; OpenMark/MacMD review velocity as same-niche traction proxies.
  4. The existential platform risk is unanalyzed: Anthropic/OpenAI/Cursor shipping rendered diff-review natively (Anthropic already adopted HTML internally; VS Code just absorbed Mermaid the same way). → Half-day scenario memo: what each could ship in two quarters, what survives each case (likely: lossless HTML + cross-agent neutrality + file-as-artifact), plus a monthly changelog monitor.
  5. Agent compliance is untested. The flagship flow assumes agents honor embedded instructions and emit well-formed ins/del — an empirical claim nobody has measured. → Scripted benchmark: 20 edit tasks × {Claude Code, Codex} × {±instructions, ±MCP}, scoring markup validity. An afternoon; directly sets the rec-5 intake priority.
  6. No retention evidence anywhere — stars and reviews measure acquisition enthusiasm; whether reviewing agent docs is a daily recurring job is the core behavioral bet. → Opt-in local-only beta counters (disclosed, aggregate) or a weekly-frequency question in interviews.
  7. The HTML-shift macro-thesis rests on one essay. → Measure it: sample recent agent output/repos for format mix; ask every interviewee what format their agents' documents actually arrive in; re-run quarterly.
  8. Distribution research covers one channel. App Store sandboxing may conflict with file-watching/localhost-Ollama/default-handler features — a channel choice that gates architecture. → One-day audit: entitlements vs feature list, Setapp criteria, MCP directories, Homebrew cask, the 10 AI-dev newsletters.
